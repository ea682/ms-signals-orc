# Hyperliquid direct receiver patch

## Goal

Receive low-latency Hyperliquid deltas from `ms-sentinel-hyperliquid` without Kafka in the hot path.

Hot path:

```text
ms-sentinel-hyperliquid
  -> POST /internal/v1/hyperliquid/deltas
  -> in-memory bounded queue
  -> direct copy dispatch executor
  -> Binance copy execution service
```

Kafka can still be used later for audit/replay from the producer side, but this receiver does not require Kafka to accept direct deltas.

## Endpoint

```http
POST /internal/v1/hyperliquid/deltas
Content-Type: application/json
X-Idempotency-Key: hyperliquid:{wallet}:{symbol}:{side}:{deltaType}:{externalId}
X-Source-Service: ms-sentinel-hyperliquid
```

The request body matches the direct payload generated by `HyperliquidPositionDeltaPayloadFactory` in `ms-sentinel-hyperliquid`.

## Performance decisions

- HTTP request does not execute copy directly.
- The controller maps and accepts into a bounded in-memory queue, then returns `202 Accepted`.
- A dedicated Hyperliquid ingest worker dispatches copy work to the existing `copyJobExecutor`.
- Direct execution bypasses the durable `copy_execution_job` polling loop when the executor has capacity.
- If direct execution is rejected or fails, the service can fallback to the existing DB job ingest path.

## Idempotency / position identity

`OperacionDto.idOperacion` is now generated from a stable position key:

```text
hyperliquid-position:{wallet}:{symbol}:{side}
```

This is intentional. Open, resize/update, and close events for the same position must use the same origin id so the existing copy engine can find and close the copied position.

The incoming `X-Idempotency-Key` is still logged and can be used for short TTL duplicate suppression if enabled.

## Config

```yaml
hyperliquid:
  direct-ingest:
    enabled: true
    queue-capacity: 20000
    worker-threads: 4
    reject-when-disabled: true
    dedupe-enabled: false
    dedupe-ttl-seconds: 30
    log-interval-ms: 10000
    fallback-db-on-direct-failure: true
```

Environment variables:

```text
HYPERLIQUID_DIRECT_INGEST_ENABLED=true
HYPERLIQUID_DIRECT_INGEST_QUEUE_CAPACITY=20000
HYPERLIQUID_DIRECT_INGEST_WORKER_THREADS=4
HYPERLIQUID_DIRECT_INGEST_REJECT_WHEN_DISABLED=true
HYPERLIQUID_DIRECT_INGEST_DEDUPE_ENABLED=false
HYPERLIQUID_DIRECT_INGEST_DEDUPE_TTL_SECONDS=30
HYPERLIQUID_DIRECT_INGEST_LOG_INTERVAL_MS=10000
HYPERLIQUID_DIRECT_INGEST_FALLBACK_DB_ON_DIRECT_FAILURE=true
```

## Loki logs

Important events:

```text
event=hyperliquid.direct_ingest.started ...
event=hyperliquid.direct_ingest.metrics ...
event=hyperliquid.direct_ingest.processed ...
event=hyperliquid.direct_copy.dispatched ...
event=hyperliquid.direct_copy.completed ...
event=hyperliquid.direct_copy.failed ...
event=hyperliquid.direct_copy.fallback_db_enqueued ...
event=http.error code=BACKPRESSURE ...
```

Per-request accepted logs are `DEBUG` to avoid excessive Loki volume.

## Notes

- `NO_CHANGE` deltas are rejected as non-copyable.
- `OPEN`, `RESIZE`, `FLIP`, and active `UPDATE` are mapped to `OperacionEvent.Tipo.ABIERTA`.
- `CLOSE`, `CLOSED` status, or zero-size resize/update are mapped to `OperacionEvent.Tipo.CERRADA`.
- Symbols without a USD/USDT suffix are normalized to `{COIN}USD`, letting the existing Binance symbol resolver translate USD to USDT.
